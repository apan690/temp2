## **Topic 33: Map Charts in Tableau**

### **Why Maps in Tableau?**

Maps are the most natural way to visualize data that has a geographic component. When your data contains countries, states, cities, or zip codes, a map immediately shows patterns that a bar chart would hide — regional clusters, geographic gaps, spatial correlations.

**When to use a map:**
- Sales performance by region or country
- Customer distribution across cities
- Store locations and their revenue
- Delivery routes or coverage areas
- Any analysis where "where" matters

---

### **Geographic Roles** ⭐ (EXAM IMPORTANT!)

Before Tableau can plot anything on a map, it needs to know which fields represent geography. This is done by assigning **Geographic Roles**.

**What is a Geographic Role?**
A tag you assign to a dimension that tells Tableau "this field contains geographic data of type X."

**How to assign:**
1. Right-click dimension in Data pane
2. **Geographic Role** →
3. Select the appropriate type

**Available Geographic Roles:**

| Role | What It Recognizes | Examples |
|------|--------------------|---------|
| **Country/Region** | Country names and codes | "India", "IN", "United States" |
| **State/Province** | State or province names | "Maharashtra", "California" |
| **City** | City names | "Mumbai", "New York" |
| **County** | County names (US-focused) | "Los Angeles County" |
| **ZIP Code / Postcode** | Postal codes | "400001", "90210" |
| **CBSA/MSA** | Metro statistical areas (US) | "New York-Newark" |
| **Area Code** | US telephone area codes | "212", "415" |
| **Latitude** | Latitude values | 19.0760 |
| **Longitude** | Longitude values | 72.8777 |

**Visual indicator:** Geographic fields show a globe icon in the Data pane.

---

### **Types of Maps in Tableau** ⭐

#### **1. Symbol Map (Default)**

**What it is:** Marks plotted as circles (or other shapes) at geographic locations. Size and color of symbol can encode measures.

**When to use:**
- Show locations of individual stores, events, customers
- Compare magnitude across cities (larger circle = more sales)
- Work well with many distinct points

**How to build:**
1. Double-click a geographic field (e.g., City)
2. Tableau automatically creates a symbol map
3. Drag measure to Size → circles scale by value
4. Drag measure to Color → color gradient shows intensity

**Example:**
```
City → automatically placed on map
SUM(Sales) on Size → bigger bubble = more sales
SUM(Profit) on Color → green = profitable, red = loss
```

**Marks Card:** Set to **Circle** (default for symbol maps)

---

#### **2. Filled Map (Choropleth Map)**

**What it is:** Geographic regions filled with color based on a measure's value. The entire polygon (state, country) is shaded.

**When to use:**
- Show which regions are performing better or worse
- Regional comparisons (country-level, state-level)
- Works best with continuous measures (revenue, population, rates)
- Fewer regions (10–50) show better — too many makes it cluttered

**How to build:**
1. Double-click a geographic field (e.g., State)
2. Tableau creates symbol map by default
3. In Marks Card dropdown → change to **Filled Map**
4. Drag measure to Color → regions fill with gradient

**Example:**
```
State → filled map
SUM(Sales) on Color → darker = higher sales
```

**Limitation:** Works at region level only, not individual points.

---

#### **3. Density Map**

**What it is:** Shows concentration of data points using a heat-based color gradient. High-density areas appear darker/brighter.

**When to use:**
- Thousands or millions of individual data points (too many for symbol map)
- Find hotspots in customer distribution, accident locations, delivery clusters
- Privacy-sensitive data where individual points shouldn't be visible

**How to build:**
1. Drag latitude/longitude to Rows and Columns
2. In Marks Card dropdown → change to **Density**
3. Adjust intensity and radius in the Marks Card

**Visual:**
```
Individual points become:
Low density → light/transparent
High density → dark/intense color
```

**Exam Note:** When asked "you have 500,000 customer locations to plot" → answer is **Density Map**.

---

#### **4. Proportional Symbol Map**

A symbol map where symbol size is explicitly proportional to a measure — technically a variation of the symbol map but commonly tested as its own type.

**How it differs from basic symbol map:**
- Size is the primary encoding (explicitly mapped to measure)
- Often combined with color for a second measure

### **Background Images (Custom Maps)**

**What it is:** Use any custom image as the map background (floor plan, stadium seating, fictional world, factory layout).

**When to use:**
- Data has custom spatial coordinates that don't match real geography
- Floor plan analysis (store layout, hospital ward)
- Any non-geographic spatial data

**How to set up:**
1. **Map** menu → **Background Images** → Select data source
2. Click **Add Image** → browse for image file
3. Define X and Y axis mapping (what your coordinates mean)
4. Set image bounds (X min/max, Y min/max)
5. Tableau plots points on image using your coordinate fields

### **Handling Map Issues** ⭐ (EXAM FAVORITE!)

#### **Problem 1: Unknown Locations**

**Symptom:** Some marks don't appear on map. Bottom right shows "X unknown" indicator.

**Causes:**
- Misspelled city/country names ("Bengaluru" vs "Bangalore")
- Abbreviations not recognized ("NY" vs "New York")
- Non-English names
- Ambiguous names (multiple cities named "Springfield")

**Solutions:**

**Fix 1: Edit Locations**
1. Click the "unknown" indicator at bottom right
2. Click **Edit Locations**
3. Manually match unrecognized values to correct locations

**Fix 2: Assign Country Context**
- Tableau may misplace "Portland" between Oregon and Maine
- Add State or Country to view to give context
- Or right-click field → Geographic Role → assign correctly

**Fix 3: Use Latitude/Longitude**
- Assign exact lat/long values
- Most accurate — no ambiguity
- Best for international data or custom locations

---

#### **Problem 2: Ambiguous City Names**

**Scenario:** "Springfield" exists in 34 US states. Tableau doesn't know which one.

**Solution:**
- Include State in the view even if you don't want to display it separately
- Drag State to Detail on Marks Card (adds context without visual change)
- Tableau uses State to disambiguate City

---

#### **Problem 3: Missing Geographic Role**

**Symptom:** Field doesn't show globe icon, treating geography as text.

**Fix:**
- Right-click field → Geographic Role → assign correct type

---

### **Latitude / Longitude Fields**

**When your data has explicit lat/long columns:**

1. Right-click Latitude column → Geographic Role → **Latitude**
2. Right-click Longitude column → Geographic Role → **Longitude**
3. Drag Latitude to **Rows** shelf
4. Drag Longitude to **Columns** shelf
5. Tableau creates a map with exact point placement

**This is the most accurate method for custom location data.**

### **Practical Map Example: Store Performance**

**Scenario:** Show store revenue across India, with profit indicated by color.

**Steps:**
1. Data has: Store_City, State, Sales, Profit
2. Assign Geographic Role: City → City, State → State/Province
3. Double-click City → symbol map appears
4. Drag Sales to Size → bigger circle = more revenue
5. Drag Profit to Color → green (profit) / red (loss) gradient
6. Drag State to Detail → resolves city ambiguity
7. Map menu → Map Layers → enable State/Province Borders

**Result:** Quick visual of top-performing stores and their profitability by location.

## **Topic 34: Data Model & Relationships** ⭐⭐

### **What Changed in Tableau 2020.2?**

Before Tableau 2020.2, combining tables meant choosing between Joins (physically merging data) or Blending (combining at viz level with limitations).

In **Tableau 2020.2**, Tableau introduced a new default way to combine tables called **Relationships** — a more flexible, intelligent approach that sits above the join layer.

**This is now the default when you drag multiple tables into the Data Source canvas.**

---

### **The Two Layers in Tableau's Data Model** ⭐

Understanding these two layers is essential to understand what relationships are and how they differ from joins.

**Visual in Data Source Tab:**
```
┌─────────────────────────────────────────┐
│          LOGICAL LAYER                   │
│  (Drag tables here to create model)      │
│                                          │
│  Orders ─────── Products                 │
│      └────────── Customers               │
│                                          │
│  Tables shown as boxes connected         │
│  by "noodle" lines = Relationships       │
└─────────────────────────────────────────┘
         Double-click a table to enter:
┌─────────────────────────────────────────┐
│          PHYSICAL LAYER                  │
│  (Traditional joins/unions live here)    │
│                                          │
│  Orders LEFT JOIN Returns (classic join) │
└─────────────────────────────────────────┘
```

**Logical Layer** = Where you define Relationships between tables
**Physical Layer** = Where traditional Joins and Unions still live (accessed by double-clicking a logical table)

---

### **What is a Relationship?** ⭐

**Definition:** A flexible connection between two tables defined by matching fields, where Tableau determines the appropriate join type automatically at query time based on what fields are used in the visualization.

**Think of it as:** Smart, deferred joining — Tableau waits to see what you're building, then queries only the needed tables at the right granularity.

**Key Concept: Relationships are NOT joins.**

---

### **Relationships vs. Joins — Critical Differences** ⭐⭐

| Aspect | Relationships | Joins |
|--------|--------------|-------|
| **When combined** | At query time (deferred) | At data source creation time (immediate) |
| **Duplicates** | Automatically prevents row duplication | Can create duplicate rows (many-to-many) |
| **NULL handling** | Preserves unmatched rows automatically | Depends on join type (inner drops them) |
| **Granularity** | Each table maintains its own granularity | Merged into single flat table |
| **Aggregation** | Aggregates happen before combining | Combines rows then aggregates |
| **Performance** | Queries only needed tables | Always queries all joined tables |
| **Flexibility** | Works across different granularities | Both tables must be at same row level |
| **Difficulty** | Easier (Tableau handles logic) | Requires explicit join type selection |

---

### **How to Create a Relationship**

**Step-by-step:**

1. **Open Data Source tab**
2. **Drag first table** to the canvas (e.g., Orders)
3. **Drag second table** next to first (e.g., Products)
4. **Relationship line (noodle)** appears automatically if matching field names detected
5. **Click the noodle** to open relationship dialog
6. **Verify or set matching fields:**
   - Orders.Product ID = Products.Product ID
7. **Set Performance Options** (optional — explained below)

**Multiple tables:**
```
Drag Orders → canvas
Drag Products → next to Orders → noodle appears
Drag Customers → next to Orders → second noodle
Drag Returns → next to Orders → third noodle

Result: Orders is the primary table, others relate to it
```

---

### **Relationship Performance Options** ⭐

When you click a relationship noodle, you can configure two optional settings that help Tableau optimize queries:

**1. Cardinality**

Describes the row-to-row relationship between tables.

| Setting | Meaning | Example |
|---------|---------|---------|
| **Many to Many** (default) | Multiple rows match on each side | Many orders per customer, many customers per order is possible |
| **Many to One** | Many rows on left, one matching on right | Many orders per customer (most common) |
| **One to Many** | One row on left, many on right | One product, many order line items |
| **One to One** | One matching row on each side | One employee, one HR record |

**2. Referential Integrity**

Tells Tableau whether every row in one table has a matching row in the other.

| Setting | Meaning |
|---------|---------|
| **Some rows match** (default) | Safe assumption — may have unmatched rows |
| **All rows match** | Every row in both tables has a match (Tableau can optimize queries) |

**Why these matter:** Correct settings allow Tableau to generate more efficient SQL and avoid unnecessary queries.

---

### **When Relationships Work Best**

✅ **Ideal for Relationships:**
- Tables at different levels of granularity (orders vs order_items vs products)
- Fact tables connecting to multiple dimension tables (star schema)
- When you want to avoid data duplication from many-to-many scenarios
- Combining tables from same or different databases

**Example — Multi-fact scenario:**
```
Orders (one row per order)
  ├── relates to Products (one row per product)
  ├── relates to Customers (one row per customer)
  └── relates to Returns (one row per returned item)

With Relationships: Each table queried at its natural granularity
With Joins: Returns would multiply order rows, inflating counts
```

---

### **When to Still Use Joins**

There are scenarios where dropping to the physical layer and using classic joins is better:

✅ **Stick with Joins when:**
- You need row-level calculations across combined tables
- You need Union (stacking rows from same-structure tables)
- Tables truly are at the same granularity and need to be flat
- You need a calculated field that combines columns from both tables pre-aggregation
- Working with spatial joins

**How to access joins within Relationships model:**
- Double-click a logical table in the canvas
- You enter the Physical Layer
- Here you can add traditional joins/unions within that logical table

---

### **Relationships vs. Joins vs. Blending — Decision Guide**

```
Tables in same data source (same connection)?
  ├── YES, and different granularities → RELATIONSHIPS (default, recommended)
  ├── YES, and same granularity → JOINS (physical layer)
  └── NO (different connections/databases)?
        ├── Need row-level detail → CROSS-DATABASE JOIN
        └── Need summary comparison → BLENDING
```

---

### **Scenario-Based Questions:**

**Q1:** You connect Orders and Products tables. When dragging them to the canvas in Tableau 2020.2+, what is created by default?
- A) Inner Join
- B) Left Join
- C) Relationship ✅
- D) Data Blend

**Why?** Tableau 2020.2+ uses Relationships by default in the logical layer. Joins still exist but require entering the physical layer.

---

**Q2:** Orders table has 1 row per order. Returns table has 1 row per returned item. You join them and SUM(Sales) doubles. What's the better approach?
- A) Use INNER JOIN instead
- B) Use Relationships ✅
- C) Use Data Blending
- D) Use a calculated field

**Why?** Relationships prevent row duplication by maintaining each table at its own granularity. The join was creating many-to-one inflation.

---

**Q3:** You want to connect Orders (SQL Server) and Targets (Excel). Which is most appropriate?
- A) Relationship
- B) Join
- C) Cross-database join or Blend ✅
- D) Union

**Why?** Different data sources — use cross-database join or blend. Relationships work within same or compatible sources.

---

**Q4:** You set Cardinality to "Many to One" on the Orders → Customers relationship. What does this tell Tableau?
- A) Every customer has many orders ✅
- B) Every order has many customers
- C) One order maps to one customer
- D) Customer data is aggregated

**Why?** Many to One = many rows on the left (Orders) match to one row on right (Customers). One customer, many orders.

---

**Q5:** You need a calculated field that multiplies Price (from Products) × Quantity (from Orders) at row level. What should you use?
- A) Relationship
- B) Physical layer Join ✅
- C) Blend
- D) LOD Expression

**Why?** Row-level calculations across tables require them to be in the same flat table — use a physical layer join. Relationships aggregate before combining.

## **Topic 35: Tableau Stories**

### **What is a Story?**

**Story** = A sequence of visualizations (story points) arranged to tell a guided data narrative.

**Think of it as:** A PowerPoint presentation built from your Tableau sheets and dashboards — each "slide" is a story point that walks the viewer through a specific insight.

**Why use stories:**
- Present findings to stakeholders in a structured narrative
- Guide viewers through data analysis step by step
- Combine multiple dashboards into one cohesive presentation
- Add context and annotations at each step

---

### **Story vs. Dashboard vs. Worksheet**

| Feature | Worksheet | Dashboard | Story |
|---------|-----------|-----------|-------|
| **Contains** | One visualization | Multiple worksheets | Multiple dashboards/sheets |
| **Purpose** | Detailed analysis | Overview, comparison | Guided narrative |
| **Interactivity** | Limited | High | Moderate |
| **Use when** | Building a view | Monitoring metrics | Presenting findings |
| **Navigation** | Filters/parameters | Actions | Story navigator (next/back) |

---

### **Story Interface**

**Components:**

**Story Navigator (top):**
- Shows all story points as thumbnails or captions
- Users click to jump between points
- Can be positioned: Top, Bottom, Left, Right

**Story Point Canvas:**
- Large area showing current story point content
- Contains a worksheet, dashboard, or blank sheet with annotations

**Caption Box:**
- Text title for each story point
- Short label (appears in navigator)

**Story Toolbar (left panel when editing):**
- **New Blank Point** — adds empty story point
- **Duplicate** — copies current point with all settings
- **Size** — set story dimensions

---

### **Creating a Story**

**Step-by-step:**

1. **New Story:** Click **New Story** tab (bottom, same as New Sheet)
2. **Set Size:** Story panel left → Size (fixed, automatic, or custom)
3. **Add Story Point:** Drag sheet or dashboard from left panel to canvas
4. **Write Caption:** Click caption box at top, type description
5. **Add Annotations (optional):**
   - **Add a caption** — highlight key takeaway text on the story point
   - Annotations appear as text overlays on the viz
6. **Add More Points:** Click **New Blank Point** or **Duplicate** in left panel
7. **Drag next sheet/dashboard** to new point canvas

**Navigation between points:**
- Left panel shows all points
- Drag to reorder

---

### **Navigator Styles**

Access: Story → Format → Navigator

| Style | Appearance | Best For |
|-------|-----------|---------|
| **Caption boxes** | Shows text captions under each thumbnail | When captions summarize each point clearly |
| **Numbers** | Shows 1, 2, 3... | Ordered sequence, no text space |
| **Arrows only** | Previous/Next arrows | Minimal UI, linear walkthrough |
| **Dots** | Small indicator dots | Clean modern style |

---

### **Story Best Practices**

✅ **Each story point = one key insight**
- Don't cram multiple messages into one point
- "Sales declined in Q3" → one point, focused on that drop

✅ **Use captions to state the insight, not describe the chart**
- Bad: "Bar chart of sales by region"
- Good: "West region underperformed target by 23%"

✅ **Progressive disclosure**
- Start with overview → drill into details in later points
- Viewer builds understanding step by step

✅ **Keep points to 5–8 maximum**
- More than 8 points loses audience attention
- Each point should earn its place

---

### **Scenario-Based Questions:**

**Q1:** You want to present quarterly performance findings to executives in a step-by-step format. Best Tableau feature?
- A) Dashboard
- B) Story ✅
- C) Multiple worksheets
- D) Parameters

**Why?** Stories are designed for guided narratives — each point presents one finding, with forward/back navigation.

---

**Q2:** A story has 7 points and you want users to jump between them non-linearly. Which navigator style works best?
- A) Arrows only
- B) Dots
- C) Caption boxes ✅
- D) Numbers

**Why?** Caption boxes show descriptive text for each point, helping users jump directly to the relevant section without going linearly.

---

**Q3:** What is the maximum recommended number of story points for audience engagement?
- A) 3
- B) 5-8 ✅
- C) 10-15
- D) No limit

**Why?** More than 8 points typically loses audience engagement. Each point should convey one focused insight.

## **Topic 36: Saving & Sharing Workbooks**

### **Tableau File Types** ⭐ (EXAM IMPORTANT!)

Understanding file types is commonly tested because choosing the wrong format is a real problem users encounter.

| Extension | Name | Contains | Use When |
|-----------|------|---------|---------|
| **.twb** | Tableau Workbook | Workbook metadata + data connections only (NO data) | Sharing workbooks when recipient has same data access |
| **.twbx** | Tableau Packaged Workbook | Everything: workbook + local data extracts + images | Sharing with someone who does NOT have data access — most common for sharing |
| **.tds** | Tableau Data Source | Connection info only (NO data, NO workbook) | Sharing a data connection definition for reuse |
| **.tdsx** | Tableau Packaged Data Source | Connection info + local extract | Sharing a data source with embedded data |
| **.tde** | Tableau Data Extract (legacy) | Older extract format (Tableau <10.5) | Legacy only — superseded by .hyper |
| **.hyper** | Hyper Extract | Modern extract format (fast, columnar) | Current extract format |
| **.tfl** | Tableau Flow | Prep Builder flow | Data preparation workflows |
| **.tfreed** | Tableau Prep Flow | Packaged Prep flow | Sharing Prep flows with embedded data |

**Most Tested:**
- **.twb** = No data included (just connection)
- **.twbx** = Data included (self-contained, shareable)
- **.hyper** = Extract file format

---

### **Tableau Publishing Destinations** ⭐

#### **Tableau Public**

**What:** Free cloud service where anyone can publish and share visualizations publicly on the web.

**Key characteristics:**
- **Free** to use
- Workbooks are **publicly visible** to everyone on the internet
- No private/confidential data should ever be published here
- Good for: Portfolio, learning, public datasets, journalism

**Limitation:** No private workbooks — everything is public.

---

#### **Tableau Server**

**What:** Tableau's on-premises server product — organizations host it on their own infrastructure.

**Key characteristics:**
- **Organization manages** the server and security
- Private and secure — within company network
- Supports user authentication (Active Directory, LDAP, SAML)
- Scheduled extract refreshes
- Row-level security possible
- Good for: Enterprise, financial data, HR data, any sensitive data

**Cost:** License-based, significant infrastructure cost

---

#### **Tableau Cloud (formerly Tableau Online)**

**What:** Tableau's cloud-hosted version of Tableau Server — Salesforce manages the infrastructure.

**Key characteristics:**
- **Salesforce/Tableau manages** infrastructure
- Private — access controlled by organization admin
- No on-premises hardware needed
- Automatic updates
- Good for: Organizations that want server capabilities without infrastructure management

**Cost:** Subscription per user

---

#### **Comparison: Public vs Server vs Cloud**

| Aspect | Tableau Public | Tableau Server | Tableau Cloud |
|--------|---------------|----------------|---------------|
| **Cost** | Free | Paid (license + infra) | Paid (subscription) |
| **Privacy** | Public only | Private | Private |
| **Hosting** | Tableau managed | Self-managed | Tableau managed |
| **Data Security** | None (public) | High (enterprise) | High (enterprise) |
| **Best For** | Portfolios, demos | Sensitive enterprise data | Cloud-first orgs |
| **Infrastructure** | None needed | Requires servers | None needed |

---

### **Export Options**

**From a worksheet or dashboard, you can export:**

**Export → Image:**
- Saves current view as PNG or BMP
- Captures visual snapshot
- Good for: Presentations, emails, reports

**Export → PDF:**
- Exports as PDF document
- Can include entire workbook or selected sheets
- Good for: Formal reports, printing

**Export → PowerPoint:**
- Exports as .pptx with Tableau views as images per slide
- Good for: Executive presentations

**Export Data:**
- **Crosstab to Excel:** Exports underlying data as Excel spreadsheet
- **Data to CSV:** Raw data extract in CSV format

**How to access:**
- **Worksheet** menu → Export
- Or right-click dashboard → Export

---

### **Saving Locally**

**File → Save:** Saves as .twb (no data)
**File → Save As:** Choose location and file name
**File → Export Packaged Workbook:** Saves as .twbx (includes data)

**Key exam scenario:**
> "You built a workbook using a local Excel file. You want to send it to a colleague who doesn't have access to that Excel file. What format?"
> **Answer: .twbx (Packaged Workbook)** — bundles the Excel data inside the file.

---

### **Tableau Public — Publishing Steps**

1. **File → Save to Tableau Public As**
2. Sign in to Tableau Public account
3. Name the workbook
4. Click Save
5. Opens in browser automatically
6. Workbook is now publicly accessible at your Tableau Public profile URL

**Note:** Tableau Public only accepts extracts (not live connections) — all data is embedded.

---

## **Topic 37: Visualization Theory — Histograms & Stacked Area Charts**

### **Histograms**

#### **What is a Histogram?**

**Histogram** = A chart that shows the **distribution** of a continuous numeric variable by dividing the range into equal-width intervals (bins) and counting how many values fall in each bin.

**Histogram vs. Bar Chart — Key Difference:**

| Aspect | Histogram | Bar Chart |
|--------|-----------|-----------|
| **X-axis** | Continuous numeric range (bins) | Discrete categories |
| **Gaps between bars** | No gaps (values are continuous) | Gaps between bars |
| **Purpose** | Show distribution/shape of data | Compare categories |
| **Data type** | One numeric measure | One dimension + one measure |

**When to use a Histogram:**
- Understand the shape of data (normal, skewed, bimodal)
- Find outliers in a numeric field
- Check if data is normally distributed
- Sales order values distribution, age distribution, salary ranges

---

#### **Understanding Bins**

**Bin** = A range/bucket that groups continuous values together.

**Example — Sales Order Values:**
```
Raw data: Orders worth $50, $120, $85, $200, $45, $310, $95, $180...

Bins of size $50:
$0-$50:    2 orders   ██
$50-$100:  5 orders   █████
$100-$150: 4 orders   ████
$150-$200: 3 orders   ███
$200-$250: 1 order    █
$250-$300: 0 orders
$300-$350: 1 order    █

Shape: Most orders cluster in $50-$150 range
```

**Bin Size Matters:**
- Too small bins → noisy, hard to see pattern (every bar looks different)
- Too large bins → over-smoothed, lose detail (everything looks same)
- Right bin size → reveals the true distribution shape

**Rule of thumb:** Start with square root of data count as number of bins. Adjust visually.

---

#### **Creating a Histogram in Tableau**

**Method 1: Show Me**
1. Select one measure field (e.g., Sales)
2. Click **Show Me** → Histogram option
3. Tableau automatically creates bins and histogram

**Method 2: Manual**
1. Right-click numeric field → **Create** → **Bins**
2. Set bin size (e.g., 100 for $100 bins)
3. Drag the new Bins field to Columns
4. Drag CNT([Field]) or Record Count to Rows
5. Chart type: Bar, no gaps

**Bin field created looks like:** `[Sales (bin)]`

### **Stacked Area Charts**

#### **What is a Stacked Area Chart?**

**Stacked Area Chart** = A line chart variation where the areas between lines are filled and stacked on top of each other, showing both individual component values and their combined total over time.

**Think of it as:** A stacked bar chart stretched over a continuous time axis.

**When to use:**
- Show how multiple categories contribute to a total over time
- Visualize part-to-whole changes over a time period
- Track cumulative trends (e.g., revenue streams building up)
- Works best with 3–6 categories maximum

---

#### **Regular vs. 100% Stacked Area Chart**

**Regular Stacked Area:**
```
Sales ($)
   ↑
   │     ▓▓▓▓▓▓▓▓▓▓▓▓  ← Technology
   │   ▓▓▓▓▓▓▓▓▓▓▓▓▓▓  ← Furniture
   │ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  ← Office Supplies
   └──────────────────→ Time

Shows: Actual values AND total
Best for: Seeing if total is growing/shrinking + category contributions
```

**100% Stacked Area:**
```
Percentage (%)
   ↑
100%│ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  ← Technology
   │ ░░░░░░░░░░░░░░░░  ← Furniture
 0%│ ████████████████  ← Office Supplies
   └──────────────────→ Time

Shows: Relative proportions only (everything = 100%)
Best for: How the MIX of categories shifts over time
```

**Key Difference:**
- Regular: See actual volume trends + mix
- 100%: See only mix changes (total volume hidden)

---

#### **Creating Stacked Area Chart in Tableau**

1. Drag **Date** to **Columns** (continuous)
2. Drag **Sales** to **Rows**
3. Drag **Category** to **Color** on Marks Card
4. In Marks Card dropdown → change to **Area**
5. Tableau stacks the areas automatically

**For 100% Stacked:**
- Apply **Percent of Total** Quick Table Calculation to the Sales pill
- Compute Using: Category (or appropriate partition)

---

#### **Stacked Area vs. Line Chart Decision**

| Use Stacked Area when | Use Line Chart when |
|-----------------------|---------------------|
| Part-to-whole over time matters | Only trend direction matters |
| Categories should be compared cumulatively | Categories should be compared independently |
| 3–6 categories maximum | Many categories (lines are clearer) |
| Showing revenue composition build-up | Showing each region's trend separately |

## **MODULE 9 SUMMARY**

### **Key Takeaways:**

✅ **Map Charts** — Symbol (points), Filled (regions), Density (heatmap for millions of points). Geographic roles must be set. Edit Locations fixes unknown entries. Use lat/long for precision.

✅ **Data Model & Relationships** — Tableau 2020.2+ default. Two layers: Logical (relationships) and Physical (joins/unions). Relationships prevent row duplication, maintain each table at its own granularity, query at runtime. Not the same as joins.

✅ **Stories** — Guided narratives built from sheets/dashboards. Each story point = one insight. Use caption boxes for non-linear navigation. Keep to 5–8 points.

✅ **Saving & Sharing** — .twb (no data), .twbx (packaged with data), .tds (connection only), .hyper (modern extract). Tableau Public = free but fully public. Server = on-premises enterprise. Cloud = hosted enterprise.

✅ **Histogram** — Distribution of continuous numeric data. Bins group values into ranges. Shape reveals normal, skewed, or bimodal patterns. Bin size affects clarity.

✅ **Stacked Area** — Regular (actual values + composition) vs. 100% (proportion only). Use Area mark type. Best with 3–6 categories.

### **Common Exam Traps:**

❌ Confusing Symbol Map with Filled Map (Symbol = points, Filled = colored regions)
❌ Thinking Relationships = Joins (they are different mechanisms with different behavior)
❌ Sending .twb to someone without data access (use .twbx instead)
❌ Confusing Histogram with Bar Chart (histogram = continuous, no gaps, one measure)
❌ Using regular stacked area when question asks about proportional change (need 100% stacked)
❌ Publishing sensitive data to Tableau Public (it's fully public!)
